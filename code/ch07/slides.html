为什么合并访问至关重要

GPU 内存事务模型（128-byte Cache Line）

未合并访问
Thread 0 → addr 0x00
Thread 1 → addr 0x40 (跨 cache line!)
Thread 2 → addr 0x80
Thread 3 → addr 0xC0
4 个事务 / warp
VS
合并访问
Thread 0 → addr 0x00
Thread 1 → addr 0x04 (同一 cache line!)
Thread 2 → addr 0x08
Thread 3 → addr 0x0C
1 个事务 / warp
float4 向量化的双重收益： 每线程 16B → 4× 更少指令 + 自然对齐到 128B 事务边界 → 最大化带宽利用率

Naive vs Tiled Matmul (N=1024)

BASELINE 
__global__ void naive_matmul(
    const float* A,
    const float* B,
    float* C, int n) {
  int row = blockIdx.y * blockDim.y + threadIdx.y;
  int col = blockIdx.x * blockDim.x + threadIdx.x;

  float sum = 0.0f;
  for (int k = 0; k < n; ++k) {
    // A 按行读 → 合并 ✓
    // B 按列读 → 跨步访问 ✗
    sum += A[row * n + k] * B[k * n + col];
  }
  C[row * n + col] = sum;
}
1.165 ms
平均耗时
OPTIMIZED 
__global__ void matmul_tiled(
    const float* A,
    const float* B,
    float* C, int n) {
  __shared__ float As[TILE][TILE];
  __shared__ float Bs[TILE][TILE];

  float sum = 0.0f;
  for (int t = 0; t < n/TILE; ++t) {
    As[ty][tx] = A[row * n + t*TILE + tx];
    Bs[ty][tx] = B[(t*TILE + ty) * n + col];
    __syncthreads();

    for (int i = 0; i < TILE; ++i)
      sum += As[ty][i] * Bs[i][tx];
    __syncthreads();
  }
}
0.367 ms
平均耗时

Shared-Memory Tiling 工作原理

Step 1: 全局→共享

Global Memory (慢, 大)
↓ 加载 tile ↓
Shared Memory (快, 小)

Step 2: 共享内存计算

As[32][32] × Bs[32][32]
×
×
每个线程累积 32 次乘加
全部在共享内存中完成

Step 3: 同步 + 写回

__syncthreads()
累积所有 tile 的结果
→
写入 Global Memory
C[row][col] = sum
关键收益： 每个 A/B 元素被 TILE 个线程复用 → 全局内存访问减少 TILE 倍（32×）→ 共享内存延迟 ~20-30 cycles vs 全局内存 ~200-400 cycles

Lookup 工作负载：数据布局重排

BASELINE 
// 随机查表: 每线程 64 步随机跳跃
__global__ void lookupNaive(
    const float* table,
    const int* indices,
    float* out, int n) {
  int index = indices[idx];
  for (int iter = 0; iter < 64; ++iter) {
    val += __ldg(table + index);
    index = advance_lcg(index);
    // LCG 随机 → cache 友好度极低
  }
}

随机跳跃 → L2 cache 几乎失效
OPTIMIZED 
// 预排布: Host 端预计算所有路径
// paths[step][idx] = table[路径值]
// 步骤维度在外，线程维度在内

__global__ void lookupOptimized(
    const float* __restrict__ paths,
    float* out, int n) {
  for (int step = 0; step < 64; ++step) {
    val += paths[step * n + idx];
    // 合并访问！相邻线程读相邻地址
  }
}

连续访问 → L2 cache line 100% 利用

核心学习目标

🎯 合并访问

测量 scalar → coalesced → float4 的带宽差距。 理解 128-byte cache line 事务模型与线程束对齐。

🎯 共享内存 Tiling

使用 shared memory tiling、async copy、tensor map 在实际工作负载中量化收益。

🎯 Cache-Locality 分析

分析 lookup-heavy 工作负载，重组数据布局 以缓解 cache-thrashing 访问模式。

🎯 量化决策

量化 transpose 和 gather/scatter 开销， 用数据驱动布局变更决策，而非凭直觉。

关键结论： Chapter 7 中最戏剧性的加速（45.4×）来自数据布局变换而非计算优化 — 内存访问模式决定性能，而非计算复杂度。

复现: python -m cli.aisp bench run --targets ch07 --profile minimal
