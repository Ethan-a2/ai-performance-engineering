DataParallel 反模式 → 直接GPU执行

单GPU上 DataParallel 强制引入不必要的 CPU-GPU 拷贝与 GIL 竞争

❌ Baseline 反模式

# 每次迭代都有 H2D 拷贝
model = nn.DataParallel(model)
for data, target in loader:
    output = model(data.to(device))
    loss = criterion(output, target)
    loss.backward()
    optimizer.step()
GIL竞争 · H2D拷贝 · 设备协调Python开销

✅ Optimized 直接执行

# 数据预置GPU，无DataParallel
# BF16 autocast加速Tensor Core
with torch.autocast("cuda", dtype=torch.bfloat16):
    output = model(inputs[step])
    loss = criterion(output, targets[step])
    loss.backward()
optimizer.step()
optimizer.zero_grad(set_to_none=True)
7.86× 加速 · 消除框架开销

流水线并行 & 张量并行

两种核心多GPU并行策略的通信模式与优化

Pipeline Parallel (1F1B)

# 朴素: 顺序前向→反向，气泡大
# 1F1B: 交错前向/反向，减小气泡
for step in range(num_microbatches):
    if step < warmup:
        output = forward(microbatch[step])
    else:
        loss.backward()          # 反向
        output = forward(...)    # 前向
    send_to_next_stage(output)

小桶压缩 vs 全缓冲压缩：分桶可减小通信粒度，但增加同步点。全缓冲减少同步但增大首字节延迟。

学习目标 & 关键收获

  • 基准测试 DataParallel / TensorParallel — 量化有/无重叠时的训练循环差异
  • NVLink 带宽与拓扑效应 — 混合本地与远程GPU时的真实带宽测量
  • NVSHMEM 流水线 — 减少主机参与GPU同步，设备侧编排
  • 对称内存池 — 跨GPU KV-cache复制与优化器状态分片的简化方案
  • 梯度融合与压缩 — 减少kernel启动次数 + 压缩通信数据量
  • NCCL 2.28 Blackwell 专属 — NVLink 5.0, C2C, TCE引擎的配置与调优
  • 分离推理架构 — Prefill/Decode分离 + 远程KV-cache复用
核心认知：通信优化不是单一缩放故事，而是框架开销移除、拓扑感知放置、计算-通信重叠、压缩量化等多维度的组合工程。
