The Problem: Launch Overhead Dominates

When kernels are small or frequent, CPU-side launch overhead becomes the bottleneck — not GPU compute.

❌ Eager Launch (Baseline)

  • Each kernel = CPU driver call
  • CUDA driver validation per launch
  • Command buffer submission overhead
  • CPU↔GPU synchronization stalls
  • ~5–10 μs overhead per launch

✅ Graph Replay (Optimized)

  • Capture workload once → replay many
  • Driver validates only during capture
  • Pre-built command buffer replayed
  • No per-launch CPU involvement
  • ~0.1 μs overhead per replay
Key Question: Not "can we capture a graph?" but "which graph or dynamic-work techniques actually reduce real runtime once correctness stays fixed?"

CUDA Graphs: Capture → Instantiate → Replay

How It Works

① Capture — Record kernel dependencies on a stream
↓
② Instantiate — Build optimized executable graph
↓
③ Replay — Launch entire graph with one CPU call
32,000 kernel launches become a single graph replay call — CPU overhead drops to near zero.

Baseline vs Optimized Code

// BASELINE: 32K separate launches for i in range(32000): kernel<<<grid, block>>>(data) // CPU call each time // OPTIMIZED: Capture once, replay 32K times cudaStreamBeginCapture(stream) kernel<<<grid, block, 0, stream>>>(data) cudaStreamEndCapture(stream, &graph) cudaGraphInstantiate(&exec, graph) for i in range(32000): cudaGraphLaunch(exec, stream) // 1 CPU call

Conditional Graph Execution

Graphs can include branches — the GPU decides which path to take, no CPU sync needed.

Baseline: Host-Side Dispatch

for i in range(5000): // Evaluate condition on GPU predicate_kernel<<<1,1< class="op">>>>(d_cond, data) // ⚠️ D2H copy = SYNC BOTTLENECK! cudaMemcpy(&h_cond, d_cond, 4, DeviceToHost) // Host decides if h_cond: expensive_kernel<<<...>>>(data) else: cheap_kernel<<<...>>>(data)

5000 × D2H sync = massive stall

Optimized: Device-Side Branching

// Capture graph with device-side branch cudaStreamBeginCapture(stream) predicate_kernel<<<1,1,0,stream>>>( d_cond, data, threshold) // Single kernel reads condition & branches conditional_dispatch_kernel<<<grid,block, 0,stream>>>(d_cond, data, N, 1.01f, 0.99f) cudaStreamEndCapture(stream, &graph) // ✅ No D2H sync — GPU decides internally for i in range(5000): cudaGraphLaunch(exec, stream)

Zero CPU↔GPU synchronization per iteration

Kernel Fusion: Fewer Launches, Fewer Memory Round-Trips

The Memory Wall Problem

Kernel A: read → compute → write back to DRAM
Kernel B: read from DRAM → compute → write back
Kernel C: read from DRAM → compute → write back

3 kernels = 3 memory round-trips (BW limited)

Fused: read → A → B → C → single write back

1 kernel = 1 memory round-trip — 2.72× faster

GPU-Resident Work Queues

Static partitioning leaves threads idle when work is uneven. Dynamic queues let fast threads steal work.

Static Distribution (Baseline)

// Each thread gets fixed indices __global__ void static_work( float* in, float* out, int N) { int idx = blockIdx.x * blockDim.x + threadIdx.x; if (idx < N) { // Fixed assignment — stragglers // block if work is uneven out[idx] = process(in[idx]); } }

CTA 0 ████████████░░ (done early, idle)
CTA 1 ████████████████████ (straggler)
CTA 2 ██████████░░░░░░ (idle)
CTA 3 ██████████████░░ (idle)

Dynamic Queue (Optimized)

// Atomic fetch-next from global counter __global__ void dynamic_work_queue( float* in, float* out, int N, int* counter) { int idx; while ((idx = atomicAdd(counter, 1)) < N) { // Fast threads grab more work out[idx] = process(in[idx]); } }

CTA 1 █████████████████ (balanced)

Dynamic Parallelism: Device-Side Launches

GPU kernels can launch child kernels — useful when workload structure is data-dependent.

Host-Side Dispatch

// CPU decides what to launch void host_dispatch(float* data, int N) { analyze<<<1,1>>>(data, &result); cudaMemcpy(&h_result, result, ...); if (h_result > threshold) { heavy_kernel<<<grid,block>>>(data); } else { light_kernel<<<grid,block>>>(data); } // ⚠️ CPU round-trip per decision }

Device-Side Launch (Dynamic Parallelism)

// GPU decides and launches children __global__ void parent_kernel( float* data, int N) { float result = analyze(data, N); if (result > threshold) { heavy_kernel<<<grid,block>>>(data); } else { light_kernel<<<grid,block>>>(data); } // ✅ No CPU involvement cudaDeviceSynchronize(); }
When to use: Dynamic parallelism shines when child-kernel count/shape depends on data. For fixed patterns, CUDA Graphs are faster (no device-side launch overhead). Requires SM 3.5+ and -rdc=true.

Uneven Workload Partitioning

When work varies across data regions, static CTA assignments create stragglers. Runtime rebalancing fixes this.

Static Partition

// Fixed tile assignment int tile_id = blockIdx.x; int start = tile_id * tile_size; int end = start + tile_size; for (int i = start; i < end; i++) { process(data[i]); // variable cost } // Some CTAs finish early, some late

Performance = slowest CTA (straggler)

Dynamic Rebalance

// Atomic work-stealing counter __shared__ int next_tile; if (threadIdx.x == 0) next_tile = atomicAdd(global, 1); __syncthreads(); while (next_tile < num_tiles) { process_tile(data, next_tile); if (threadIdx.x == 0) next_tile = atomicAdd(global, 1); __syncthreads(); } // ✅ Fast CTAs take more work

Performance ≈ average CTA (balanced)

Requires: -rdc=true (relocatable device code) for device-side atomics on global counters. Uses helper_*.cu host/device glue files.

Key Takeaways

When Graphs Help

  • Workload is stable enough to capture
  • Launch overhead > compute time
  • Many small kernels in sequence
  • Repetitive steady-state patterns

When Fusion Helps

  • Memory-bound pipelines
  • Multiple kernels touching same data
  • Intermediate results not needed
  • DRAM bandwidth is the bottleneck

When Dynamic Dispatch Helps

  • Work depends on runtime data
  • Irregular batch sizes
  • Uneven per-element cost
  • Static partitioning → stragglers

Decision Framework

1. Is workload stable/repetitive? → CUDA Graphs (4× speedup) 2. Are kernels memory-bound? → Kernel Fusion (2.7× speedup) 3. Is work data-dependent? → Dynamic Parallelism 4. Are batches irregular/uneven? → Work Queues (4.75× speedup) 5. None of the above? → Eager launches may be fine.
Bottom line: The chapter separates "graphs help" from "graphs help on a workload stable enough to benefit." The work-queue target keeps the chapter from being only about graph replay.
