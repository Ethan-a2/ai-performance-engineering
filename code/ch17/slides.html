The Problem

路由不再是白板架构，而是要付租金的工程

❌ 核心问题

  • 静态路由导致队列不均衡
  • KV Cache 迁移开销主导延迟
  • Prefill/Decode 阻塞式交接造成 GPU 空闲
  • 统一 MoE 路由忽略拓扑局部性

✅ 正确的问题

"哪种路由器、队列策略和交接机制
能在完整 Prefill/Decode 路径下
真正改善 TTFT、TPOT 或吞吐？"

Baseline vs Optimized

两条路径的对比

🐢 Baseline Path

  • 静态或最小自适应路由
  • 保守的 Prefill/Decode 交接
  • 更多阻塞行为
  • 易于推理，但队列不均衡和 KV 迁移代价高昂
  • 拓扑无关的统一 MoE 路由

🚀 Optimized Path

  • 拓扑感知 / 遥测感知路由
  • 分离式 Prefill/Decode 减少空闲时间
  • 批处理交接 / 重叠执行
  • 通过共享 harness 度量，可与 kernel/memory 优化对比
  • 拓扑感知专家路由

Benchmark B

MoE Router — 拓扑感知 vs 统一路由

Baseline: 统一路由

拓扑无关，token 均匀分布到所有 island

4 islands × 16 experts = 64 experts

大量跨岛远程传输，高 remote-transfer fraction

Optimized: 拓扑感知路由

优先路由到本地 island 的专家

共享权重保证输出一致性 — 加速来自局部性改善

5.07×
Speedup — 减少远程传输，提升局部性

设计约束

  • 所有专家共享相同权重
  • 路由决策不改变输出张量语义
  • 可度量的加速来自 locality，而非数学变化
  • 256 batch × 64 seq，模拟远程 EP 传输

Multi-GPU Variants

分离式推理的多种优化变体

变体 优化焦点 运行方式 关键差异
prefill_decode_disagg 基础分离 单 GPU 单体 vs 分离基准对比
..._ttft_multigpu TTFT 优化 torchrun Prefill/Decode pool 分离，最小化首 token 延迟
..._overlap_multigpu 重叠执行 torchrun Prefill/Decode 段重叠，减少 idle bubble
..._batched_multigpu 批处理交接 torchrun per-group 批处理替代 per-request 阻塞
..._tpot_long_multigpu 长输出 TPOT torchrun 长上下文场景下的每 token 延迟优化

📌 torchrun 多 GPU 变体默认 50/50 split（偶数 world size），可通过 --prefill-ranks 覆盖（如 2P1D）

实现细节

  • _PIPELINE_STAGE_COUNT = 4
  • Stage 边界：[(0,1), (2,3), (4,5), (6,6)]
  • 模型：Linear → GELU → Linear → GELU → ...
  • 优化版重叠 Prefill/Decode 段
  • 减少 idle bubble，提升 GPU 利用率
