The Core Problem

推理系统级优化的关键问题：什么编排变化真正降低了延迟、提高了吞吐？

Monolithic Inference

  • ✗ Prefill 和 Decode 互相阻塞
  • ✗ KV 缓存管理简单但昂贵
  • ✗ 队列排空策略效率低
  • ✗ 单机推理吞吐受限

Disaggregated Inference

  • ✓ Prefill/Decode 分离到不同池
  • ✓ NVLink 池化 KV 缓存
  • ✓ 连续批处理调度
  • ✓ 投机解码加速生成

关键问题：KV 移动和调度开销被包含在内时，哪些编排变化能真正降低延迟或提高吞吐？

Monolithic vs Disaggregated Inference

Baseline: 单体推理

class BaselineInferenceMonolithicBenchmark:
    """单一服务同时处理 prefill 和 decode"""
    
    def benchmark_fn(self):
        # Prefill: 处理整个 prompt (compute-bound)
        logits, kv_cache = self.model(self.prompt)
        next_token = logits[:, -1].argmax(dim=-1)
        
        # Decode: 逐 token 生成 (memory-bound)
        for _ in range(self.num_tokens):
            logits, kv_cache = self.model(
                next_token.unsqueeze(0), kv_cache
            )
            next_token = logits[:, -1].argmax(dim=-1)

问题：Prefill (compute-bound) 和 Decode (memory-bound) 共享同一 GPU，互相阻塞，资源利用率低

Optimized: 分离推理

Prefill Pool

高算力 GPU 组
处理 prompt 编码
生成 KV cache

↓ KV Transfer ↓

Decode Pool

高带宽 GPU 组
逐 token 解码
读取 KV cache

连续批处理 (Continuous Batching)

动态调度 vs 静态批处理：4.16x 加速

Baseline: 静态批处理

class BaselineContinuousBatchingBenchmark:
    """固定批次成员，padding 浪费"""
    
    def __init__(self):
        # static=True: 批次内所有请求
        # 必须同时完成
        super().__init__(
            dynamic=False,
            multi_gpu=False,
        )

问题

  • 短请求等待最长请求完成
  • Padding 浪费算力
  • GPU 利用率低

52.96 ms

Optimized: 动态连续批处理

class OptimizedContinuousBatchingBenchmark:
    """请求完成即释放，新请求立即加入"""
    
    def __init__(self):
        # dynamic=True: 流式调度
        # TTFT 感知优先级
        super().__init__(
            dynamic=True,
            multi_gpu=False,
        )

优势

  • 请求完成即出队
  • 新请求零等待加入
  • GPU 持续满载

# Host-staged prefill/decode 分离
class HostStagedPrefillDecodeDisaggBenchmark:
    def benchmark_fn(self):
        # Phase 1: Prefill (compute-bound)
        prefill_out = self.prefill_model(prompt)
        kv_cache = prefill_out.kv_cache
        
        # KV transfer: prefill GPU → decode GPU
        kv_on_decode = kv_cache.to(decode_device)
        
        # Phase 2: Decode (memory-bound)
        for _ in range(num_tokens):
            tok = self.decode_model(tok, kv_on_decode)

多 GPU 分离要点

  • Prefill 和 Decode 可在不同 GPU 上
  • KV Cache 通过 NVLink/PCIe 传输
  • 可叠加投机解码进一步加速
  • 支持 --simulate-network 模拟模式

放置策略模拟器

多节点推理的拓扑感知调度

@dataclass
class PlacementConfig:
    """放置策略的可调参数"""
    prefill_tp_size: int       # Prefill TP 并行度
    prefill_span_nodes: bool   # Prefill 跨节点?
    decode_tp_size: int        # Decode TP 并行度
    decode_span_nodes: bool    # Decode 跨节点?

# 核心启发式策略:
# 1. MoE expert 并行限制在 NVLink 域内
# 2. Prefill 用 TP, Decode 可降到 TP=1
# 3. Session 路由保持 KV cache 亲和性
# 4. 跨节点通信惩罚计入延迟

模拟器目标

  • 量化 NVLink 域内 vs 跨节点开销
  • 比较 TP/EP/PP 组合
  • 评估 KV 迁移代价
  • 发现尾延迟风险

关键约束

  • Expert 并行 ≤ NVLink 域
  • Decode 粘附 KV 所在节点
  • 跨节点集体通信有惩罚
  • 支持 B200/B300/GB200
